# UX Audit - 2026-07-19

## Product Activation Pass

- Replaced the fresh-workspace zero-metric Dashboard with one outcome: connect Telegram, receive a real customer message, and send the first manual reply.
- Telegram onboarding now opens a guided first-run dialog, establishes an Inbox baseline before enabling the bot link, polls only while the page is visible, ignores internal samples, detects new messages on existing conversations, and opens the exact conversation.
- Conversation messages show persisted delivery status. Activation completes only at provider `SENT` or `DELIVERED`; queued and failed messages remain visibly incomplete. The completion panel moves into view immediately and does not retain contradictory queued feedback.

- Removed the Settings notification tab and its eager API request because no notification dispatcher consumes those preferences. This supersedes the earlier accessibility-only treatment of those switches.
- All activation copy is complete in EN, RU, ES, FR, DE, and PT. Focused browser coverage protects baselining, sample exclusion, plan preservation, provider acceptance, canonical website inbound evidence, and the pre-activation Dashboard.

## Method

- Inventoried visible links, buttons, inputs, selects, tabs, menus, dialogs, and other interactive roles with Playwright.
- Exercised navigation, mobile menus, language selection, form validation, representative filters and dialogs, and page/nested scrolling.
- Monitored page and console errors, failed requests, HTTP error responses, route settlement, and global horizontal overflow.
- Compared desktop and mobile behavior; sampled wheel response on both page and nested scroll regions.

## Results And Fixes

- No uncaught page/console errors, failed browser requests, HTTP error responses, or global horizontal overflow occurred in the inventoried states.
- Sampled page and nested regions changed scroll position by the first animation frame; the reported delayed-scroll behavior was not reproduced.
- Public navigation, mobile menu behavior, language option semantics, authentication form validation, and demo route transitions behaved consistently in the exercised states.
- Homepage niche selectors changed content but did not expose selection state. They now use a labeled group and `aria-pressed`, with exactly one selected option.
- Demo Pipeline Kanban/List controls now expose their active state with `aria-pressed`.
- Invalid email addresses now keep the OTP action disabled and show localized inline validation on blur in all six locales.
- Restoring a saved theme no longer creates a server/client hydration mismatch.
- Fourteen visible mobile actions across Dashboard, Inbox, Knowledge, and Billing now meet the 44px target minimum. Native checkbox/radio inputs remain paired with their larger visible labels.
- All five Settings notification switches now derive accessible names from their visible localized labels.
- Dashboard activity uses localized labels for known system action codes and preserves stored titles only for unknown actions.
- Knowledge V2 list requests failed only in the metadata-less local `tsx` runtime because numeric query values stayed strings. Explicit DTO types now protect all paginated Knowledge V2 controller queries; focused validation passed `29/29`, and the affected live local endpoints returned `200`.
